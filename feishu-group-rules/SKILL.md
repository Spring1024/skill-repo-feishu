# 飞书群聊协作规范 (Feishu Group Rules)

> **版本**: v2.1  
> **日期**: 2026-07-12  
> **适用范围**: 飞书群聊中所有 Hermes Agent 的协作规范

---

## 一、 消息类型与格式要求

你在群聊中发送的**每一条**消息，都必须以明确的类型标签开头。标签格式必须严格一致，且**必须位于消息文本的第一行**，严禁使用以下五种以外的任何自定义标签：

1. **任务类**：当需要安排其他智能体或用户执行任务时。
   - 格式：`===📋任务📋===`
2. **回复类**：当汇报任务执行结果时。
   - 格式：`===✅️回复✅️===`
3. **通知类**：当仅用于广播信息、无需特定对象回复时。
   - 格式：`===📣通知📣===`
4. **异常类**：当任务执行失败、报错或无法完成时。
   - 格式：`===❌异常❌===`
5. **询问类**：当需要询问其他智能体关于任务执行进展，需要用户做决定时。
   - 格式：`===❓询问❓===`

## 二、 混合消息格式

一条消息可以包含**多个标签**，每个标签对应不同类型的信息。

### 格式规则

```
===✅️回复✅️=== 回复内容
===📋任务📋=== 任务内容
===📣通知📣=== 通知内容
===❌异常❌=== 异常内容
===❓询问❓=== 询问内容
```

### 示例

**示例 1：回复 + 任务**

```
===✅️回复✅️=== 任务已完成
- 任务: 查询用户数据
- 结果: 共查询 1000 个用户，成功 980 个，失败 20 个

===📋任务📋=== 请波比处理失败的 20 个用户
- 任务: 处理失败用户数据
- 具体要求:
  1. 分析失败原因
  2. 重新查询失败用户
  3. 返回处理结果
- 截止时间: 20 分钟内
- 优先级: P1
```

**示例 2：回复 + 通知**

```
===✅️回复✅️=== 任务已完成
- 任务: 查询用户数据
- 结果: 成功查询 1000 个用户

===📣通知📣=== 数据已导出到 /data/users.csv
```

**示例 3：回复 + 异常**

```
===✅️回复✅️=== 部分任务已完成
- 任务: 查询用户数据
- 结果: 成功查询 800 个用户

===❌异常❌=== 剩余 200 个用户查询失败
- 失败原因: 数据库连接超时
- 已重试: 3 次
- 建议: 请联系 DBA 检查数据库状态
```

**示例 4：任务 + 通知**

```
===📋任务📋=== 请波比补充技术选型细节
- 任务: 补充技术选型细节
- 具体要求:
  1. 确认前端框架
  2. 确认后端框架
  3. 确认数据库选型
- 截止时间: 30 分钟内

===📣通知📣=== 此任务优先级为 P1（重要）
```

### 标签顺序规则

1. **回复类**（===✅️回复✅️===）必须放在最前面
2. **任务类**（===📋任务📋===）放在第二位
3. **询问类**（===❓询问❓===）放在第三位
3. **通知类**（===📣通知📣===）放在第四位
4. **异常类**（===❌异常❌===）放在最后

### 注意事项

- 每条消息**至少包含一个标签**
- 每条消息**最多包含四个标签**
- 标签之间用**空行**分隔
- 每个标签的内容必须**独立成段**
- 标签**必须位于消息的第一行**


## 二、 默认消息类型判定与意图过滤

- **意图过滤**：如果用户的消息仅为简单的问候、确认（如"收到"、"好的"）、闲聊或无实质内容的表情，无需添加任何标签，直接进行自然语言回复或忽略。
- **任务判定**：只有当用户发送了包含实质性需求或指令的消息，且**没有**显式包含上述四种类型标签时，才默认视为**任务类**消息。收到此类消息时，你需要主动拆解任务，并使用 `===📋任务📋===` 标签向相关智能体或用户下达指令。

## 三、 回复与通知的 @ 规则

1. 接收 **回复类消息**：接收到@消息，并且为回复类消息时，分为三种情况：Ⅰ这条回复消息不涉及任务时，无需再次回复**回复消息** ；Ⅱ回复消息涉及任务分配时，需要执行，等待执行完成后再回复；Ⅲ基于回复消息的内容，需要重新分配新的任务时，此时使用**任务类**消息，进行回复。
2. **回复类消息**：必须使用 `@` 标签精准指派当前任务的发起者（用户或智能体），并简要复述原任务的关键特征，将结果告知对方。
3. **询问类消息**：必须使用 `@` 标签精准指派需要询问的对象（用户或智能体）。
4. **通知类消息**：严禁使用 `@` 标签，直接将内容发送到群聊即可。

## 四、 防死循环与熔断机制 (Anti-Loop & Circuit Breaker)

为防止智能体之间陷入无限互相 @ 的死循环，需要安装 `distributed-anti-loop-plugin.py` 插件，同时你必须严格遵守以下规则：

- **禁止无意义重试**：如果同一个任务连续失败超过 3 次，必须立即停止重试。
- **监控规则**：在任意 15 分钟的滚动时间窗口内，如果你与某一个特定智能体互相 `@` 回复的次数**超过 20 次**。
- **熔断动作**：立即停止对该智能体的 `@` 回复。
- **降级处理**：
  - 如果任务是**用户发起**的 → 将消息发送给**该用户**（私聊）
  - 如果任务是**智能体发起**的 → 将消息发送给**群主**（@群主）进行人工介入

## 五、 Bot 身份标识

每个 Bot 在发送消息时，必须在消息开头添加身份标识，格式为：`[Bot:名称]`

**示例**：
```
[Bot:芙蓉] ===📋任务📋=== 请波比帮我查一下数据
```

**身份标识规则**：
- 使用 Bot 的**名称**（如芙蓉、波比）
- 必须用方括号包围
- 紧跟在消息类型标签之前

## 六、 避免话题嵌套（做出如下修改）

**核心规则**：在群聊中回复时，必须使用**新消息**格式，而不是"回复消息"格式。

### 问题描述

飞书"回复消息"功能会创建**话题（Thread）**，导致消息脱离群聊主时间线。当一条消息被多次使用"回复"类型的消息引用时，会自动形成话题，从而脱离群聊。

### 解决方案

修改Hermes 飞书适配器，强制指定 `parent_id` 为触发机器人的消息 ID，防止话题嵌套。

**修改文件**：`/feishu/adapter.py`

**修改内容**：`_process_inbound_message` 方法中原先的

```python
# 第一处修改
thread_id = getattr(message, "thread_id", None) or getattr(message, "root_id", None) or None
reply_to_message_id = (
    getattr(message, "parent_id", None)
    or getattr(message, "upper_message_id", None)
    or getattr(message, "root_id", None)
    or None
)
reply_to_text = await self._fetch_message_text(reply_to_message_id) if reply_to_message_id else None

# 第二处修改
chat_id = getattr(message, "chat_id", "") or ""
chat_info = await self.get_chat_info(chat_id)
...
thread_id=thread_id,
channel_prompt=self._resolve_channel_prompt(chat_id, thread_id or None),


# 第三处修改
trigger_message_id = getattr(message, "message_id", None) or None
reply_to_message_id = trigger_message_id
...
normalized = MessageEvent(
    ...
    reply_to_message_id=reply_to_message_id,
    reply_to_text=reply_to_text,
)
```

**修改为**：

```python
# 第一处修改
# 强制指定 parent_id 为触发机器人的消息 ID，防止话题嵌套
# 永远只引用 @ 机器人的那条消息，而不是透传历史消息链中的 ID
trigger_message_id = getattr(message, "message_id", None) or None
reply_to_message_id = trigger_message_id

# 第二处修改
chat_id = getattr(message, "chat_id", "") or ""
Extract thread_id from message (parent_id for topic threads, None for DM/single chat)
thread_id = getattr(message, "parent_id", None) or getattr(message, "root_id", None)
chat_info = await self.get_chat_info(chat_id)
...
thread_id=thread_id,
channel_prompt=self._resolve_channel_prompt(chat_id, thread_id),

# 第三处修改
trigger_message_id = getattr(message, "message_id", None) or None
reply_to_message_id = trigger_message_id
reply_to_text = None
...
normalized = MessageEvent(
    ...
    reply_to_message_id=reply_to_message_id,
    reply_to_text=reply_to_text,
)
```

**效果**：在群聊中回复时，永远只引用 @ 机器人的那条消息，不会形成话题嵌套。


## 七、 超时询问与心跳检测机制

### 7.1 超时询问流程

当任务派发后，如果接收方长时间无响应，发送方需要按照以下流程进行询问：

| 时间点 | 询问类型 | 消息格式 | 处理方式 |
|--------|----------|----------|----------|
| **30 分钟** | 第一次询问 | `===❓询问❓===` | 发送询问，等待 10 分钟 |
| **40 分钟** | 第二次询问 | `===❓询问❓===` | 发送询问，等待 10 分钟 |
| **50 分钟** | 第三次询问 | `===❓询问❓===` | 发送询问，等待 10 分钟 |

**第一次询问格式**（30 分钟）：

```
===❓询问❓=== 任务进度询问
- 任务: [任务描述]
- 已用时: 30 分钟
- 请回复: 是否仍在处理中？
```

**第二次询问格式**（40 分钟）：
```
===❓询问❓=== 任务进度询问（第二次）
- 任务: [任务描述]
- 已用时: 40 分钟
- 已询问: 1 次
- 请回复: 是否仍在处理中？
```

**第三次询问格式**（50 分钟）：
```
===❓询问❓=== 任务进度询问（第三次）
- 任务: [任务描述]
- 已用时: 50 分钟
- 已询问: 2 次
- 请回复: 是否仍在处理中？
- 警告: 若不再回复，将标记为失败
```

### 7.2 心跳检测规则

**核心规则**：
- 接收方收到询问后**必须回复**
- 如果回复 → 重置计时器，异常次数清零，任务继续
- 如果未回复 → 异常次数 +1，继续等待下一次询问

**正常回复格式**（任务继续）：
```
===✅️回复✅️=== 任务仍在处理中
- 任务: [任务描述]
- 当前进度: [具体进度]
- 预计完成时间: [时间]
- 状态: 正常
```

**失败回复格式**（主动放弃）：
```
===❌异常❌=== 任务执行失败
- 任务: [任务描述]
- 失败原因: [具体原因]
- 已尝试: N 次
- 建议: [解决方案]
```

### 7.3 异常判定规则

| 异常次数 | 处理方式 |
|----------|----------|
| 1 次 | 继续等待，发送下一次询问 |
| 2 次 | 继续等待，发送第三次询问 |
| 3 次 | **标记失败**，通知发起人 |

**标记失败后通知发起人格式**：
```
===✅️回复✅️=== 任务执行结果
- 任务: [任务描述]
- 状态: 失败
- 失败原因: [具体原因]
- 已尝试: 3 次
- 建议: [解决方案]
- 联系人: [Bot 名称] 的管理员
```

### 7.4 心跳健康检查（可选）

为了防止 Agent 假死（回复了但任务没进展），可以定期发送心跳检查：

**心跳消息格式**（每 10 分钟）：
```
===❓询问❓=== 心跳检查
- 任务: [任务描述]
- 当前进度: [具体进度]
- 状态: 正常/异常
```

**心跳回复格式**：
```
===✅️回复✅️=== 心跳正常
- 任务: [任务描述]
- 当前进度: [具体进度]
- 状态: 正常
- 最后更新时间: [时间]
```

**心跳超时规则**：
- 如果 Agent 连续 3 次心跳超时（30 分钟）
- 标记为假死
- 发送最终询问
- 若仍无回复 → 标记失败

## 八、 协作超时机制

### 8.1 超时时间规定

| 类型 | 超时时间 | 处理方式 |
|------|----------|----------|
| **响应超时** | 5 分钟 | 发送第一次提醒 |
| **执行超时** | 30 分钟 | 发送超时询问 |
| **心跳超时** | 10 分钟 | 标记为假死 |

### 8.2 超时处理流程

```
任务派发
  ↓
开始计时
  ↓
5 分钟 → 发送第一次提醒（📣通知）
  ↓
30 分钟 → 发送超时询问（第一次）
  ↓
40 分钟 → 发送超时询问（第二次）
  ↓
50 分钟 → 发送超时询问（第三次）
  ↓
60 分钟 → 标记失败 → 通知发起人
```

## 九、 异常分类与处理

### 9.1 异常类型

| 异常类型 | 触发条件 | 处理方式 |
|----------|----------|----------|
| **超时类** | Agent 长时间无响应 | 发送询问 → 标记失败 |
| **执行失败类** | Agent 收到任务但执行失败 | 发送失败通知 → 建议方案 |
| **资源不足类** | CPU/内存/磁盘/API 配额不足 | 发送资源报告 → 扩容建议 |
| **网络异常类** | DNS/连接/SSL 错误 | 发送网络报告 → 检查配置 |
| **假死类** | 回复了但任务没进展 | 标记假死 → 最终询问 |

### 9.2 异常消息格式

**超时类异常**：
```
===❌异常❌=== 任务超时
- 任务: [任务描述]
- 超时时间: 30 分钟
- 已询问: 3 次
- 建议: 请联系 [Bot 名称] 的管理员
```

**执行失败类异常**：
```
===❌异常❌=== 任务执行失败
- 任务: [任务描述]
- 失败原因: [具体原因]
- 已尝试: N 次
- 建议: [解决方案]
```

**资源不足类异常**：
```
===❌异常❌=== 资源不足
- 任务: [任务描述]
- 不足资源: [CPU/内存/磁盘/API 配额等]
- 预估需求: [具体数值]
- 建议: [扩容方案]
```

**网络异常类**：
```
===❌异常❌=== 网络异常
- 任务: [任务描述]
- 异常类型: [DNS 解析失败/连接超时/SSL 证书错误等]
- 目标地址: [URL/IP]
- 建议: [检查网络配置/防火墙/代理等]
```
